# Phase 2b~3가 "별도 서비스"인 이유 — 쉽게 설명

> **목적**: "Phase 2b~3(오케스트레이터·자동 에이전트 등)는 별도 서비스로 구축해야 하고, 이 레포는 계약·프롬프트 소스만 제공한다"는 말의 의미를 쉽게 풀어서 설명합니다.  
> **중요**: **실행 엔진**은 Cursor Cloud Agent가 할 수 있습니다. Slack 연동 시 사람이 @Cursor 호출하거나, Cloud Agents API로 프로그램에서 호출하면 됩니다. "별도 서비스"는 **그걸 붙이는 Slack 앱·라우팅·상태 관리**를 말합니다.

---

## 1. 한 줄 요약

- **이 레포(cursor-agent-boilerplate)** = **설계도 + 대본**. "누가 무엇을 하고, 어떤 형식으로 말을 주고받을지" **정의만** 해 둔 곳.
- **실행 엔진** = **Cursor Cloud Agent**. Slack에서 @Cursor 로 호출하거나, [Cloud Agents API](https://cursor.com/docs/cloud-agent/api/overview)로 호출하면, 실제 LLM·코드 실행·슬랙 응답은 Cursor가 처리합니다. **직접 LLM 런타임을 만들 필요는 없습니다.**
- **Phase 2b~3에서 "별도로" 필요한 것** = **Slack 앱(포지션별)·라우팅·(선택) 오케스트레이터**. "어느 채널/어느 순간에 Cursor Cloud Agent를 어떤 역할(프롬프트)로 부를지" 연결해 주는 부분. 각 포지션을 **Slack 앱**으로 두면, 그 앱이 해당 포지션 채널에서 @Cursor 또는 API로 Cursor Cloud Agent를 그 역할 컨텍스트(이 레포의 프롬프트)로 호출하게 할 수 있습니다.

---

## 2. 비유로 이해하기

| 비유 | 이 레포 | Phase 2b~3에서 구축하는 것 |
|------|---------|-----------------------------|
| **연극** | 대본·역할 설명·무대 이동 순서(스크립트) | 실제 배우가 무대에 서서 연기하고, 스태프가 조명·음향을 돌리는 **공연 자체** |
| **레시피** | 요리책(재료·순서·완성 형태) | 주방·가스레인지·요리사가 요리하는 **실제 요리 과정** |
| **회사** | 직무 정의서·업무 매뉴얼·보고 형식 | 그 매뉴얼대로 일하는 **사람·시스템**이 실제로 업무를 수행하는 것 |

이 레포는 **"이렇게 하세요"라고 적어 둔 문서**이고,  
Phase 2b~3는 **그 문서를 읽고 실제로 동작하는 프로그램(오케스트레이터, 에이전트 런타임 등)** 을 만드는 단계입니다.

## 3. "계약·프롬프트 소스"만 제공한다는 뜻

### 3.1 이 레포에 **있는** 것 (정의·문서)

- **계약**  
  - 역할이 누구인지 (PO, 디자이너, FE, BE, QA).  
  - 어떤 이벤트를 누가 누구에게 보내는지 (`spec_completed`, `pr_opened` 등).  
  - 메시지 안에 어떤 필드가 들어가야 하는지 (JSON 스키마).  
  - 상태가 어떻게 바뀌는지 (TODO → IN_PROGRESS → DONE).  
  → [AGENT_MESSAGE_PROTOCOL.md](AGENT_MESSAGE_PROTOCOL.md), [AGENT_MEMORY_SCHEMA.md](AGENT_MEMORY_SCHEMA.md) 등에 적혀 있음.

- **프롬프트 소스**  
  - 각 역할(포지션)이 **무슨 말을 해야 하는지**, **어떤 규칙으로 판단할지** 적어 둔 텍스트.  
  - 예: `pm/AGENTS.md`, `backend/.cursor/rules/`, `pm/slack-agent-prompt.md` 등.  
  → 나중에 **에이전트를 돌리는 프로그램**이 이 파일들을 읽어서 "시스템 프롬프트"로 씀.

정리하면, 이 레포는  
**"누가, 무엇을, 어떤 형식으로 할지"** 와 **"각 역할에게 줄 지시문(프롬프트)"** 를 **문서로만** 갖고 있다는 뜻입니다.  
그래서 **"계약·프롬프트 소스만 제공"** 이라고 부릅니다.

### 3.2 실행 엔진 = Cursor Cloud Agent (Slack 연동 시)

**질문**: "실행 엔진은 Slack이랑 연동되면 Cursor Cloud Agent가 하는 거 아니야?"

**맞습니다.**  
Slack과 Cursor를 연동해 두면:

- **사람이 슬랙에서 @Cursor** 를 부르면 → **Cursor Cloud Agent**가 그 스레드 맥락을 보고 작업하고, 결과를 슬랙에 돌려줍니다. 이때 **실제 LLM 호출·코드 실행·레포 클론·PR** 같은 건 전부 Cursor가 처리합니다. 즉 **실행 엔진 = Cursor Cloud Agent**입니다.
- **포지션별로** 쓰는 방법:
  - **채널로 구분**: #silo-po, #silo-design, #silo-be, #silo-fe, #silo-qa 처럼 포지션별 채널을 두고, 그 채널에서 @Cursor 할 때 "이 채널은 PO 역할이야" 같은 **컨텍스트(이 레포의 pm/slack-agent-prompt.md 등)** 를 메시지에 넣거나, 채널 설명에 적어 두면 됩니다. Cursor 앱은 하나만 써도 됩니다.
  - **Slack 앱을 포지션별로**: @PO-Bot, @Design-Bot, @BE-Bot 처럼 **포지션마다 Slack 앱(봇)** 을 두고, 각 봇이 해당 포지션 채널에서만 반응하게 할 수도 있습니다. 그 봇이 **Cursor Cloud Agents API**를 호출할 때, 이 레포의 해당 포지션 프롬프트(AGENTS.md, slack-agent-prompt.md)를 컨텍스트로 넘기면, **실행 엔진은 여전히 Cursor Cloud Agent**이고, Slack 앱은 "누구 역할로 부를지"만 연결해 주는 역할입니다.

정리하면, **직접 LLM 런타임을 만들 필요는 없고**, Slack 연동 + Cursor Cloud Agent(또는 API)로 실행 엔진을 씁니다.  
각 포지션은 **Slack 채널** 하나로 갈 수도 있고, **Slack 앱(봇)** 하나씩으로 갈 수도 있습니다.

### 3.3 그럼에도 "별도로" 필요한 것 (이 레포에 없는 것)

실행은 Cursor가 하므로, **우리가 만드는 쪽**은 다음 정도입니다.

- **Slack 앱(들)**  
  - 포지션별 채널에서 **@Cursor** 를 쓰는 방식이면: Cursor 공식 Slack 연동만 하면 되고, **추가 Slack 앱 개발은 선택**.  
  - 포지션별 **Slack 봇**(@PO-Bot, @BE-Bot 등)으로 쓰려면: 그 봇이 **Slack 이벤트 수신 → Cursor Cloud Agents API 호출(이 레포의 해당 포지션 프롬프트 전달) → 결과를 슬랙에 포스팅** 하도록 **Slack 앱(서버)** 을 하나씩(또는 하나에 여러 봇) 만듦. 이 레포에는 그 **연동 로직·서버 코드**가 없음.

- **오케스트레이터 (선택)**  
  - "스펙 완료됐을 때 자동으로 디자인 채널에 할 일 넣고 Cursor 호출" 같은 **자동 전이**를 하려면, **이벤트를 받아서 상태를 바꾸고, 적절한 채널/봇으로 Cursor를 부르는** 작은 서비스가 필요. 이걸 오케스트레이터라고 부르고, 이 레포에는 없음.  
  - 전부 사람이 슬랙에서 @Cursor 부르는 방식이면 오케스트레이터 없이도 가능.

- **에이전트 메모리 (선택)**  
  - 스레드/태스크별 맥락을 **저장**해 두려면 DB나 저장소가 필요. 스키마는 이 레포 [AGENT_MEMORY_SCHEMA.md](AGENT_MEMORY_SCHEMA.md)에 정의만 있고, 실제 저장소는 별도 구축.

- **GitHub/CI 연동 (선택)**  
  - PR 생성·CI 상태·배포·롤백을 **자동**으로 하려면, Cursor Cloud Agent가 GitHub를 쓰는 부분에 더해, 웹훅·API를 받아서 오케스트레이터나 Slack에 알려 주는 연동 코드가 필요. 이 레포에는 "이렇게 연동하라"는 계약만 있음.

**요약**:  
- **실행 엔진** = **Cursor Cloud Agent** (Slack @Cursor 또는 Cloud Agents API).  
- **포지션** = **Slack 채널** 또는 **Slack 앱(봇)** 으로 구분 가능.  
- **"별도 서비스"** = Cursor 말고 우리가 붙이는 **Slack 앱(봇)·(선택) 오케스트레이터·(선택) 메모리·GitHub/CI 연동**.

## 5. 자주 나오는 질문

**Q. 실행 엔진은 Slack이랑 연동되면 Cursor Cloud Agent가 하는 거 아니야?**  
**A.** 맞아요. **실행 엔진 = Cursor Cloud Agent**입니다. Slack에서 @Cursor 를 부르거나, Cloud Agents API로 호출하면, LLM·코드 실행·슬랙 응답은 Cursor가 처리합니다. 우리가 별도로 LLM 런타임을 만들 필요는 없습니다.

**Q. 각 포지션별로 Slack 앱으로 되겠지?**  
**A.** 네. 포지션을 나누는 방법은 두 가지입니다.  
- **채널로만 구분**: #silo-po, #silo-be, #silo-qa 처럼 포지션별 채널을 두고, 같은 **Cursor 앱 하나**만 쓰면서, 각 채널에서 @Cursor 할 때 "PO 역할로 해줘" 같은 문구나 이 레포의 해당 포지션 프롬프트를 맥락으로 넣어 주면 됩니다.  
- **Slack 앱(봇)을 포지션별로**: @PO-Bot, @BE-Bot, @QA-Bot 처럼 **포지션마다 Slack 앱**을 두고, 각 앱이 해당 채널에서 메시지를 받으면 **Cursor Cloud Agents API**를 호출할 때 이 레포의 그 포지션 프롬프트(pm/, backend/, qa/ …)를 넘기면, 실행은 Cursor가 하고 앱은 "어떤 역할로 부를지"만 연결해 줍니다.  
둘 다 "각 포지션 = Slack에서 하나의 진입점(채널 또는 앱)" 구조로 가면 됩니다.

**Q. 그럼 이 레포만 있으면 슬랙에서 에이전트가 알아서 일해요?**  
**A.** "에이전트가 일한다"는 부분은 **Cursor Cloud Agent**가 합니다. 이 레포는 **그걸 어떤 역할(프롬프트)로 부를지, 어떤 형식으로 말을 주고받을지**만 정의해 둔 곳입니다.  
- **사람이 슬랙에서 @Cursor** 부르는 방식이면, Cursor Slack 연동만 하면 되고 추가 개발은 거의 없음.  
- **봇이 자동으로 Cursor를 부르게** 하려면, 포지션별 Slack 앱(봇)이 **Cursor Cloud Agents API**를 호출하는 서버를 별도로 만들면 됩니다. 그때 이 레포의 해당 포지션 프롬프트를 API에 넘기면 됩니다.

**Q. "계약·프롬프트 소스"는 뭘 하는 데 쓰나요?**  
**A.**  
- **Slack에서 @Cursor** 쓸 때: 각 채널/포지션에서 "이 역할로 해줘"라고 할 때 참고할 **지시문·규칙**을 이 레포(AGENTS.md, slack-agent-prompt.md 등)에서 복사하거나 링크로 씀.  
- **Slack 앱(봇) + Cursor API** 쓸 때: 봇이 Cursor Cloud Agents API를 호출할 때 **시스템 프롬프트**로 이 레포의 해당 포지션 파일 내용을 넘김.  
- **에이전트끼리 주고받는 메시지 형식**은 AGENT_MESSAGE_PROTOCOL.md 에 정의돼 있으니, 오케스트레이터나 봇을 만들 때 그걸 따르면 됨.

**Q. Phase 2b~3를 하려면 뭘 만들어야 하나요?**  
**A.**  
- **실행 엔진**은 Cursor Cloud Agent를 쓰면 되므로 **새로 안 만듦**.  
- **만들 수 있는 것**: (1) **포지션별 Slack 앱(봇)** — 해당 채널에서 메시지 수신 → Cursor Cloud Agents API 호출(이 레포의 해당 포지션 프롬프트 전달) → 결과 슬랙 포스팅. (2) **(선택) 오케스트레이터** — 이벤트에 따라 상태 전이·자동으로 특정 채널/봇에 Cursor 호출 트리거. (3) **(선택) 메모리·GitHub/CI 연동** — 스레드 맥락 저장, PR·CI·배포 연동.  
이 레포는 **계약·프롬프트 소스**만 제공하고, 위 (1)(2)(3)은 **별도 저장소·서버**에서 구축합니다.

## 6. 정리

- **실행 엔진** = **Cursor Cloud Agent**. Slack 연동 시 @Cursor 또는 Cloud Agents API로 호출하면, LLM·코드·슬랙 응답은 Cursor가 처리합니다. 별도 LLM 런타임을 만들 필요 없습니다.

- **포지션별 구분** = **Slack 채널** (#silo-po, #silo-be, …) 하나로 할 수도 있고, **Slack 앱(봇)** (@PO-Bot, @BE-Bot, …)을 포지션마다 둘 수도 있습니다. 각 채널/봇에서 Cursor를 **해당 포지션 프롬프트(이 레포)** 로 호출하면 됩니다.

- **"Phase 2b~3는 별도 서비스로 구축해야 한다"**  
  = 실행은 Cursor가 하므로, **별도로 만드는 것**은 **Slack 앱(봇)·(선택) 오케스트레이터·(선택) 메모리·GitHub/CI 연동** 같은 **연결·라우팅·자동화**용 코드/서버입니다. 이 레포에는 그 구현이 없습니다.

- **"이 레포는 계약·프롬프트 소스만 제공한다"**  
  = **역할 정의, 메시지 형식(계약), 각 역할용 지시문(프롬프트)** 를 **문서로만** 제공합니다. Cursor를 **어떤 역할로, 어떤 맥락으로** 부를지는 이 레포를 보고, Slack 앱·오케스트레이터는 별도 프로젝트에서 구현합니다.
