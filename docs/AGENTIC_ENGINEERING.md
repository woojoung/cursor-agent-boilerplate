# 에이전틱 엔지니어링 — Cursor 에이전트가 똑똑·효율적으로 작업하기 위한 원칙

> 이 문서는 [에이전틱 엔지니어링 시대의 생존 스킬 9가지](https://news.hada.io/topic?id=27104)(GeekNews)를 이 보일러플레이트에 적용한 요약입니다.  
> **목적**: 에이전트와 사람이 협업할 때 "위임 패러독스"를 줄이고, **구현 전 설계·완료 정의·실패 복구·맥락 유지**를 명확히 해 더 효율적으로 작업하게 합니다.

---

## 1. 지금까지 이 레포에서 된 작업 (요약)

- **포지션별 구조**: backend, frontend-web, frontend-mobile, qa, infra, pm, design — 각각 `.cursor/`(rules, skills, commands, agents), AGENTS.md, .cursor-plugin.
- **기술 스택 정합성**: 백엔드(Java/Spring), 프론트 웹(Next.js), 모바일(Flutter)에 맞게 .cursor 내용 검증 완료. [docs/SETUP_REPOS.md](SETUP_REPOS.md)
- **실무 적용**: 플러그인 또는 복사로 실무 레포에 적용. Slack 봇 연동·팀 워크플로우는 [INTEGRATION.md](INTEGRATION.md).
- **확장성**: 포지션별 TOOLS_AND_MCP.md(도구·MCP), [EXTENSIBILITY.md](EXTENSIBILITY.md). Agentation, Figma/Stitch MCP 예시.
- **새 레포 세팅**: 에이전트 대화창에서 `/setup-repo` 또는 "새 레포 세팅해줘"로 해당 포지션 프로젝트 생성 + .cursor·AGENTS.md 복사.

---

## 2. 생존 스킬 9가지 → 이 보일러플레이트에 적용할 부분

아래는 GeekNews 기사의 9가지 스킬을 **Cursor 에이전트가 더 똑똑·효율적으로 동작하도록** 적용한 내용입니다.  
에이전트는 이 원칙을 **규칙·스킬·AGENTS.md**를 통해 참고할 수 있도록 합니다.

### ① 분해 능력 (Decomposition)

- **기사 요지**: "회원가입 만들어줘" 한 번에 던지면 원하는 게 안 나올 확률이 높다. **구현 전 5분 생각·소크라틱 대화**가 4시간을 아낀다.
- **적용**: 
  - **spec → plan → task** 순서를 이미 강제하고 있음. 
  - 에이전트 규칙에 추가: **큰 요청을 받으면 구현 들어가기 전에 요구·엣지 케이스·범위를 한두 턴 질문으로 정리**하고, 사용자 확인 후 plan/task 단위로 진행.
  - PROMPTS·명령에서 "한 번에 하나의 Phase·한 가지 검증 목표"로 나누어 요청하도록 안내.

### ② 컨텍스트 설계 (Context Architecture)

- **기사 요지**: AGENTS.md만이 아니라 **코드/폴더 구조**가 잘 잡혀 있으면 에이전트가 컨텍스트를 파악하는 속도가 다르다. 플랫 디렉터리보다 feature·레이어 단위가 유리.
- **적용**: 
  - 이미 **rules(coding-style)** 에서 backend=domain/application/adapter, frontend-web=app/components/lib, frontend-mobile=presentation/data 구조를 요구함.
  - 에이전트 규칙에 명시: **플랫 디렉터리·이름만으로 역할이 안 드러나는 폴더 지양**. feature·역할 단위로 디렉터리 구성 권장.

### ③ 완료 정의 (Definition of Done)

- **기사 요지**: **에이전트의 "완료"와 사람의 "완료"가 다르다.** 타입만 세팅하고 비즈니스 로직은 빈 껍데기로 끝낼 수 있음.
- **적용**: 
  - **plan.md·task.md**에 "완료 시 확인할 것" 항목을 넣도록 명령·스킬에서 유도 (이미 plan-from-spec 등에서 Phase별 검증 언급).
  - 규칙에 추가: **작업 완료 선언 전에 DoD 체크** — (예) 빌드/테스트 통과, 리뷰 체크리스트 항목 충족, AC(수용 조건) 만족. 사용자에게 "다음 중 미충족 항목이 있으면 알려주세요" 형태로 요약 제시.

### ④ 실패 복구 (Failure Recovery Loop)

- **기사 요지**: **같은 프롬프트로 재시도하는 건 벽에 같은 방향으로 머리 박는 것.** 실패를 3가지(컨텍스트 부족, 방향 오류, 구조적 충돌)로 분류하면 처방이 달라진다. "Must NOT Have" 가드레일이 무한루프를 끊었다.
- **적용**: 
  - 규칙·스킬에 추가: **재시도 시 "왜 실패했는지" 먼저 분류** — (1) 컨텍스트 부족 → AGENTS.md·관련 파일·스펙 보강 후 재요청, (2) 방향 오류 → 요구·제약을 명확히 하고 방향 수정, (3) 구조적 충돌(의존성·네이밍 충돌) → 영향 범위 정리 후 단계적 수정.
  - **Must NOT Have** (금지 사항)를 spec·plan 또는 rules에 명시해 두면, 에이전트가 그 경계를 넘지 않도록 유도. (이미 coding-style·security 등에 "금지" 섹션 있음.)

### ⑤ 관찰 가능성 (Observability)

- **기사 요지**: "이상한데… 그냥 두자"가 가장 비싼 판단. 예광탄 전략 + 블루프린트로 처음 적용하는 기술은 빠르게 그려보고, **관찰 가능성이 신뢰를 만들고 위임을 가능하게** 한다.
- **적용**: 
  - 규칙에 추가: **의심스러운 동작·불명확한 요구는 "그냥 두지 말고"** 사용자에게 짧게 요약해 확인 요청. 새 기술·새 패턴 도입 시 "예광탄(최소 동작 예시) 먼저 제안" 후 확장.
  - 리뷰·plan 단계에서 "검증 가능한 지표·단계"를 넣도록 유도.

### ⑥ 메모리 설계 (Memory Architecture)

- **기사 요지**: 3일 연속 작업하면 매일 15분 맥락 설명. **세션 종료 시 맥락 추출 → 다음 세션 5초 복원.** CLAUDE.md를 git에 체크인해 팀이 공유.
- **적용**: 
  - **AGENTS.md**가 이미 "팀·에이전트가 보는 맥락" 역할. 
  - docs 또는 규칙에 안내: **긴 작업·멀티 세션 시** 세션 끝에 "이번 세션에서 한 일·다음에 할 일·결정 사항"을 AGENTS.md 또는 `docs/context/` 같은 곳에 2~3문장으로 남기면 다음 세션 복원에 유리. 
  - (선택) 팀에서 CLAUDE.md 또는 CONTEXT.md를 레포에 두고 주기적으로 갱신하도록 규칙에 한 줄 추가.

### ⑦ 병렬 관리 (Parallel Orchestration)

- **기사 요지**: 에이전트는 묻지 않고 자기 판단으로 진행하므로 **사전 설계가 더 중요**하다. 여러 세션을 돌릴 때도 설계가 있으면 흐트러지지 않는다.
- **적용**: 
  - **spec·plan을 먼저 고정**하는 흐름이 이미 있음. 
  - 규칙에 한 줄: **병렬로 여러 작업을 돌릴 때는 공통 spec·plan·브랜치 전략을 먼저 맞춰 두고**, 각 세션에서 "이 task만" 범위를 명시해 요청하도록 안내.

### ⑧ 추상화 계층 설계 (Abstraction Layering)

- **기사 요지**: 반복 루틴을 스킬·명령으로 만들면 20분이 2분으로. **컴파운딩 엔지니어링** — 앞 세션이 뒤에 복리로 영향을 준다.
- **적용**: 
  - 이미 **.cursor/commands, .cursor/skills**로 반복 작업을 추상화해 둠. 
  - 규칙·docs에 안내: **자주 쓰는 지시·검증 루틴은 스킬 또는 명령으로 추가**하면 다음부터 일관되고 빠르게 실행됨. (EXTENSIBILITY.md와 연계.)

### ⑨ 감각 (Taste)

- **기사 요지**: AI 출력은 60~70점. **구현 자동화가 될수록 설계·판단·감각의 중요성이 오른다.** 차별화는 나머지 20%에서 나온다.
- **적용**: 
  - 규칙에 추가: **에이전트 제안은 "초안·후보"로 두고**, 최종 설계·네이밍·UX·보안 판단은 **사람이 리뷰·승인**한다. 코드 리뷰·plan 승인·DoD 확인은 사람이 하는 단계로 명시.

---

## 3. 에이전트가 지킬 원칙 (한 줄 요약)

- **분해**: 구현 들어가기 전에 요구·엣지 케이스·범위를 정리하고 확인받는다.
- **컨텍스트**: feature·레이어 단위 구조를 유지하고, AGENTS.md를 갱신한다.
- **완료 정의**: 완료 선언 전 DoD(빌드·테스트·AC·리뷰)를 체크하고 사용자에게 요약한다.
- **실패 복구**: 같은 프롬프트 반복 대신, 실패 유형(컨텍스트/방향/구조)을 나누고 그에 맞는 처방을 한다. Must NOT Have를 준수한다.
- **관찰 가능성**: 의심스러우면 "그냥 두지 말고" 확인 요청. 새 기술은 예광탄 제안 후 확장.
- **메모리**: 긴 작업·멀티 세션 시 맥락 요약을 AGENTS.md 등에 남긴다.
- **병렬**: 여러 세션 시 spec·plan·범위를 먼저 맞춘 뒤 task 단위로 요청한다.
- **추상화**: 반복 루틴은 스킬·명령으로 추가해 재사용한다.
- **감각**: 최종 설계·판단·승인은 사람이 한다. 에이전트 출력은 검토·보정 대상으로 둔다.

이 원칙은 **.cursor/rules** 의 main 또는 별도 rule에서 참조하고, **AGENTS.md**에 "에이전틱 엔지니어링 원칙: docs/AGENTIC_ENGINEERING.md 참고" 한 줄을 넣으면 에이전트가 로드해 활용할 수 있습니다.
