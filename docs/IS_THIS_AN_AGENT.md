# 지금까지 작업한 걸 "에이전트"라고 부를 수 있나?

> **목적**: 이 보일러플레이트가 **에이전트**라고 불릴 수 있는지, 어떤 의미에서 "에이전트"인지 짧게 정리합니다.

---

## 1. 결론: **에이전트(구성)라고 부를 수 있다**

- **Cursor 기준**: 이 레포에는 **서브에이전트(subagent)** 가 포지션·역할별로 정의돼 있습니다.  
  예: `planner`, `code-reviewer`, `security-reviewer`, `spec-writer`, `design-reviewer` 등.  
  각각 `.cursor/agents/*.md` 에 이름·설명·호출 시 할 일이 적혀 있고, Cursor가 이걸 **에이전트**로 인식해 위임·실행합니다.
- **역할 단위**: 포지션(backend, pm, design, …) 하나하나는 **한 역할을 가진 에이전트 설정 묶음**입니다.  
  rules + commands + skills + agents + AGENTS.md + slack-agent-prompt.md 가 합쳐져서, "이 역할의 에이전트가 이렇게 행동한다"를 정의합니다.
- **Slack·비전 관점**: docs에 정리한 대로, 이 정의를 **Slack + Cursor Cloud Agent**나 **포지션별 Slack 앱**에서 불러서 쓰면, "PO 에이전트", "BE 에이전트"처럼 **역할별 에이전트**로 동작하게 설계돼 있습니다.

그래서 **"에이전트(에이전트 구성/보일러플레이트)"** 라고 부르는 것이 맞고,  
- **개별 에이전트**: `.cursor/agents/` 안의 planner, code-reviewer 등 + 포지션별 역할 정의  
- **에이전트 보일러플레이트**: 포지션별로 그런 에이전트를 쓰기 위한 규칙·명령·스킬·프롬프트·프로토콜을 한 레포에 모아 둔 것  

이라고 보면 됩니다.

---

## 2. "에이전트"라고 하기 애매한 부분이 있다면

- **실행 주체**: 지금은 **Cursor(와 사람)** 이 실행 주체입니다.  
  "에이전트가 스스로 슬랙에 말하고 일한다"는 건 Cursor Cloud Agent나 별도 봇을 붙였을 때 나오는 동작이지, 이 레포 **파일만** 놓고 보면 "에이전트 정의·설정"에 가깝습니다.
- **그래서 쓰는 표현**  
  - **"에이전트 보일러플레이트"** / **"에이전트 구성"** / **"역할별 에이전트 설정"** → 정확함.  
  - **"에이전트다"** 만 말하면, "실행하는 쪽은 Cursor/봇이고, 이 레포는 그 에이전트의 **정의·소스**"라고 한 줄만 더 해주면 헷갈리지 않습니다.

---

## 3. 한 줄 요약

- **에이전트라고 부를 수 있다.**  
  Cursor 서브에이전트 + 포지션별 역할 정의 + (선택) Slack에서 부를 때의 프롬프트까지 포함한 **에이전트용 보일러플레이트/설정 묶음**이다.
- GitHub·Cursor 플러그인에 올릴 때 **이름**에 "agent"를 넣어도 (예: `cursor-agent-boilerplate`) 문제 없고,  
  설명에 "역할별 에이전트 설정·보일러플레이트"라고 적어 두면 의미가 잘 전달됩니다.
