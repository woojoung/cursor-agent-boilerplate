# 실무 레포 적용 · Slack 봇 연동 가이드

> 이 문서의 목적: (1) **각 포지션별로** 이 보일러플레이트를 **실무 작업 레포**에서 어떻게 쓰는지, (2) **플러그인 또는 복사**로 세팅된 실무 레포들이 **Slack 봇**과 함께 **실제 IT 회사 팀 프로젝트**처럼 어떻게 유기적으로 돌아가는지 설명합니다.

---

## 1. 각 포지션에서 Cursor 플러그인 사용 가능 여부

**가능합니다.**  
이 레포의 **모든 포지션 폴더**에 `.cursor-plugin/plugin.json`이 있으므로, 각 포지션 폴더는 **플러그인 소스**로 사용할 수 있습니다.

| 포지션 폴더      | 플러그인 이름 (plugin.json)              | 용도 요약                    |
|------------------|------------------------------------------|-----------------------------|
| `backend/`       | cursor-agent-boilerplate                 | 백엔드 Java/Spring·TDD·리뷰 |
| `frontend-web/`  | cursor-agent-boilerplate-frontend-web    | Next.js·a11y·컴포넌트       |
| `frontend-mobile/` | cursor-agent-boilerplate-frontend-mobile | Flutter·위젯·ViewModel      |
| `qa/`            | cursor-agent-boilerplate-qa              | 시나리오·회귀·버그템플릿    |
| `infra/`         | cursor-agent-boilerplate-infra           | IaC·Terraform·런북         |
| `pm/`            | cursor-agent-boilerplate-pm              | 스펙·태스크 분해·리스크     |
| `design/`        | cursor-agent-boilerplate-design          | 디자인시스템·토큰·a11y      |

## 2. 실무 레포에서 사용하는 방법 (플러그인 vs 복사)

실제로 **팀이 작업하는 레포**(예: `product-a-backend`, `product-a-frontend`)에 이 보일러플레이트를 적용하는 방법은 두 가지입니다.

### 방법 A: 플러그인으로 사용

- **Cursor에서 "플러그인 폴더"로 추가하는 경우**  
  - 이 레포를 클론한 뒤, **실무 레포를 Cursor로 Open** 합니다.  
  - Cursor 설정에서 **플러그인을 "로컬 폴더"로 추가**할 수 있다면, **해당 포지션 폴더 경로**를 지정합니다.  
  - 예: 실무 레포가 백엔드라면 `.../cursor-agent-boilerplate/backend` 를 플러그인 폴더로 지정.  
  - 그러면 해당 실무 레포에서 작업할 때 **그 포지션의 rules·commands·skills·agents**가 플러그인으로 적용됩니다.
- **마켓플레이스에 등록한 경우**  
  - 포지션별로 Cursor Marketplace에 플러그인을 등록해 두었다면, 실무 레포를 연 상태에서 **해당 포지션 플러그인을 설치**하면 같은 효과를 냅니다.  
  - (현재는 이 레포 내부에만 플러그인 정의가 있으므로, 배포 시 포지션 폴더 단위로 publish 하면 됩니다.)

**정리**:  
- **각 포지션별 레포** = 이 레포의 **포지션별 폴더**(backend, frontend-web, …)를 말할 수 있고,  
- **각 포지션별 레포에서 플러그인 사용** = 그 **포지션 폴더를 플러그인 소스로 지정**하면 됩니다.  
- 실무 레포는 "내 코드만 있는 레포"이고, "규칙·명령·스킬·에이전트"는 **이 보일러플레이트의 해당 포지션 폴더**에서 플러그인으로 가져옵니다.

### 방법 B: 복사로 사용

- 실무 레포 **루트**에, 사용할 포지션 폴더의 다음 내용을 **복사**합니다.  
  - `.cursor/` (rules, commands, skills, agents 전부)  
  - `AGENTS.md`  
- 필요 시 `.cursor-plugin/` 도 복사해 두면, 그 실무 레포 자체가 같은 포지션의 플러그인 구조를 가진 "하나의 플러그인 소스"처럼 동작할 수 있습니다.  
- 복사 후 팀 규칙에 맞게 **규칙·스킬·에이전트 내용만 수정**해 사용하면 됩니다.

**어떤 방법을 쓸지**  
- **플러그인**: 규칙을 "한곳(보일러플레이트 또는 내부 역할 정의 레포)에서만 관리"하고, 여러 실무 레포에서 동일 설정을 쓰고 싶을 때 유리합니다.  
- **복사**: 실무 레포마다 **완전히 독립된 설정**을 두고, 레포별로 커스터마이즈를 많이 할 때 유리합니다.

## 3. Slack 봇 + 실무 레포: 팀 프로젝트처럼 유기적으로 돌아가는 흐름

아래는 **플러그인 또는 복사로 실무 레포에 세팅이 끝난 뒤**, **Slack 봇**과 **실무 레포**가 어떻게 엮여서 실제 IT 회사 팀 프로젝트처럼 움직일 수 있는지 단계별로 정리한 것입니다.  
부족한 부분이 있으면, 이 흐름을 기준으로 채널·봇·웹훅만 추가하면 됩니다.

### 3.1 구성 요소 정리

| 구성 요소 | 역할 |
|----------|------|
| **역할 정의 레포 (이 보일러플레이트)** | 포지션별 Rules·Commands·Skills·Agents·AGENTS.md·플러그인 정의. "어떤 역할이 어떤 규칙으로 일하는지"의 단일 소스. |
| **실무 레포** | 팀/프로젝트/포지션별 실제 작업 공간 (예: product-a-backend, product-a-frontend, product-a-qa-docs). Cursor에서 Open 하고, 위 규칙을 **플러그인 또는 복사**로 적용. |
| **Slack** | 일일 업무 소통·알림·승인·이슈 공유. 포지션별 채널 + BOSS/리더 채널. |
| **Slack 봇** | "역할 정의"는 **역할 정의 레포(이 보일러플레이트)** 에서 로드. 실무 레포는 **Git 웹훅/이벤트**로만 구독해 **알림·상태 반영**용으로 사용. |

### 3.2 팀 워크플로우 (한 사이클)

실제 회사에서 "기획 → 개발 → 리뷰 → QA → 배포 → 슬랙 공유"가 도는 것처럼, 아래 순서로 유기적으로 이어질 수 있도록 설명합니다.

1. **기획·스펙 (PM)**  
   - PM은 **스펙·이슈**를 담당 레포 또는 이슈 트래커에서 작성합니다.  
   - Cursor는 **pm** 포지션 플러그인/복사가 적용된 레포에서 `/spec-template`, `/task-breakdown`, `/risk-check` 등으로 스펙 초안·태스크 분해·리스크를 정리합니다.  
   - **Slack**: PM 채널 또는 BOSS 채널에 "이번 스프린트 스펙 정리됐어요, 이슈 링크 공유" 식으로 올립니다. 봇이 "스펙 요약"을 역할 정의 레포의 문구/템플릿을 참고해 올리도록 할 수 있습니다.

2. **개발 (백엔드·프론트 웹·프론트 모바일 등)**  
   - 각 포지션 담당자는 **자기 실무 레포**를 Cursor에서 Open 하고, 해당 포지션의 **플러그인(또는 복사된 .cursor)** 으로 spec → plan → task → 구현·리뷰·MR까지 진행합니다.  
   - **Slack**:  
     - MR 생성·리뷰 요청 시 **실무 레포의 Git 웹훅**을 Slack 봇이 받아서, 해당 프로젝트 채널에 "MR 올라왔음, 리뷰어 @멘션" 같이 알립니다.  
     - 봇이 "역할 정의 레포"에서 가져온 **리뷰 체크리스트·MR 설명 템플릿**을 댓글이나 스레드로 달아 주면, 팀이 같은 기준으로 리뷰할 수 있습니다.

3. **코드 리뷰**  
   - 리뷰어는 Cursor에서 `/review-code`, `/code-reviewer` 등으로 리뷰하고, MR에 코멘트를 남깁니다.  
   - **Slack**: MR 승인/거절·리뷰 완료 시 웹훅으로 알림. BOSS 채널에는 "중요 MR 머지됨" 정도만 요약해도 됩니다.

4. **QA**  
   - QA는 **qa** 포지션 플러그인/복사가 적용된 레포(또는 테스트 전용 레포)에서 `/scenario-from-spec`, `/test-plan`, `/bug-template`으로 시나리오·테스트 계획·버그 리포트를 작성합니다.  
   - **Slack**:  
     - 버그 등록·테스트 완료를 QA 채널에 공유.  
     - 실무 레포에서 "release branch 배포 완료" 같은 이벤트가 웹훅으로 오면, 봇이 "배포 완료, QA 산타 전환 가능" 식으로 알릴 수 있습니다.

5. **인프라·배포 (선택)**  
   - 인프라 담당은 **infra** 포지션 플러그인/복사로 `/plan-from-spec`, `/review-infra`, `/runbook-draft` 등을 사용합니다.  
   - **Slack**: 배포 시작/완료/롤백 시 웹훅으로 알림. 장애 시 런북 링크를 봇이 올리도록 할 수 있습니다.

6. **디자인 (선택)**  
   - 디자인은 **design** 포지션으로 컴포넌트 규격·토큰·접근성 리뷰를 하고, **Slack**에 디자인 리뷰 요청·완료를 공유할 수 있습니다.

7. **BOSS·의사결정**  
   - BOSS(또는 팀 리더)는 **Slack의 BOSS 채널**에서:  
     - 스펙·우선순위 승인,  
     - MR 머지·릴리스 타이밍,  
     - 장애 대응·롤백 결정  
     등을 합니다.  
   - 봇은 "역할 정의 레포"의 규칙(예: MR 설명 형식, 스펙 상태)을 참고해, BOSS 채널에 **요약·체크리스트·승인 요청** 문구를 보여 줄 수 있습니다.

### 3.3 Slack 봇이 "연동"하는 방법 (보완 포인트)

- **역할 정의 (규칙·프롬프트)**  
  - Slack 봇의 "지식" 또는 "시스템 프롬프트"는 **이 보일러플레이트 레포(역할 정의 레포)** 를 소스로 합니다.  
  - 주기적으로 레포를 pull 하거나, 웹훅으로 "역할 정의 레포 main 갱신됨" 이벤트를 받아 봇이 최신 규칙·템플릿을 반영하도록 하면 됩니다.

- **실무 레포 이벤트 (알림·상태)**  
  - 각 **실무 레포**에는 Git(GitLab/GitHub) **웹훅**을 설정합니다.  
  - 웹훅 페이로드: push, MR 생성/머지/리뷰, issue 생성/클로즈, release 태그 등.  
  - Slack 봇 서버가 이 웹훅을 받아서:  
    - 해당 프로젝트·포지션에 맞는 **Slack 채널**에 알림을 보내고,  
    - 필요 시 "역할 정의 레포"에서 가져온 **MR 설명 템플릿·리뷰 체크리스트·버그 템플릿**을 스레드에 붙여 주면, 팀이 같은 기준으로 일하게 됩니다.

- **부족할 수 있는 부분과 보완**  
  - **채널 매핑**: "레포 A = #proj-a-backend, 레포 B = #proj-b-frontend" 같은 매핑 테이블을 봇 설정에 두고, 웹훅 올 때마다 채널을 선택해 알림을 보냅니다.  
  - **이슈 트래커 연동**: Jira/Linear 등 이슈가 생성·완료되면 봇이 Slack에 요약해 주면, "스펙 → 이슈 → 개발 → QA" 흐름이 Slack에서 한눈에 보입니다.  
  - **문서화 규칙**: 역할 정의 레포의 AGENTS.md·rules에 "MR 시 체크리스트", "스펙 상태 변경 시 Slack에 한 줄 공유" 같은 규칙을 넣어 두고, 봇이 그 문구를 Slack에 재사용하면 "역할별 히스토리·문서화"가 자연스럽게 지켜집니다.

## 4. 역할–채널 매핑 (오케스트레이터 설정)

팀/사일로별 Slack 채널 ↔ 포지션 매핑은 **[docs/AGENT_MESSAGE_PROTOCOL.md](AGENT_MESSAGE_PROTOCOL.md)** §4를 단일 소스로 사용합니다.  
예: `#silo-a-po` → pm, `#silo-a-design` → design, `#silo-a-fe` → frontend-web, `#silo-a-be` → backend, `#silo-a-qa` → qa, `#silo-a-boss` → 인간(전략·최종 승인).

## 5. 슬랙·Boss 고도화: 포지션별 프롬프트 템플릿과 유기적 협업

Boss가 슬랙 봇으로 업무를 지시하거나, 각 포지션 에이전트가 **서로 유기적으로** 동작하게 하려면 아래를 적용할 수 있습니다.

### 5.1 포지션별 "자주 쓰는 지시" 템플릿 (슬랙 → Cursor 연동)

각 포지션 폴더에 **PROMPT_TEMPLATES.md**가 있습니다. 슬랙 봇이 "해당 포지션 채널"에 올릴 때 아래처럼 **지시 문구 템플릿**을 붙이면, 담당자가 Cursor에 그대로 붙여 넣어 에이전트를 바로 활용할 수 있습니다.

| 포지션 | 슬랙에서 봇이 안내할 수 있는 지시 예시 (Cursor에 붙여넣기용) |
|--------|-------------------------------------------------------------|
| **PM** | "스펙 초안 만들어줘. [기능명]에 대해 목표·요구·AC 포함해줘." / "위 스펙 기준으로 태스크 분해해줘." / "리스크·의존성 점검해줘." |
| **디자인** | "[컴포넌트명] 컴포넌트 규격 문서 초안 만들어줘." / "지금 [대상] 디자인 리뷰해줘." |
| **QA** | "이 스펙 기준으로 테스트 시나리오 도출해줘." / "이번 변경 범위로 테스트 계획 수립해줘." / "버그 리포트 템플릿 채워줘." |
| **인프라** | "인프라 요구 기준으로 plan 만들어줘." / "Terraform/파이프라인 리뷰해줘." / "런북 초안 작성해줘." |
| **백엔드** | "스펙 기준으로 plan·task 만들어줘." / "이 MR 리뷰해줘." / "MR 설명 초안 만들어줘." |
| **프론트(웹/모바일)** | 동일하게 스펙→plan→task, 리뷰, MR 초안 등. |

- **역할**: Boss가 슬랙에서 "이번 주 PM은 스펙 초안, 백엔드는 plan 작성해줘"라고 지시할 때, 봇이 **해당 포지션 채널**에 위 문구(또는 PROMPT_TEMPLATES.md의 "자주 쓰는 프롬프트")를 스레드로 달아 주면, 담당자는 Cursor에서 그 문구만 복사해 에이전트에 넣으면 됩니다.
- **에이전트 고도화**: 각 포지션 에이전트는 **PROMPT_TEMPLATES.md**와 **`/start`** 명령을 읽고, "뭐 할 수 있어?" 같은 모호한 요청에도 **옵션을 제시한 뒤 역으로 질문**하므로, 해당 포지션을 잘 모르는 사람도 전문가처럼 결과물을 요청할 수 있습니다.

### 5.2 유기적 협업을 위한 핸드오프 포맷

포지션 간 **산출물이 다음 포지션으로 넘어갈 때** 형식을 맞추면, 에이전트가 맥락을 이해하고 잘 동작합니다.

| 이전 포지션 | 다음 포지션 | 핸드오프에 포함할 것 (슬랙·이슈·MR에 명시) |
|-------------|-------------|---------------------------------------------|
| **PM → 개발** | 백엔드/프론트 | 스펙 문서 링크(spec.md 또는 이슈), 목표 한 줄, AC 목록, 우선순위(P0/P1). |
| **PM → QA** | QA | 스펙/요구사항 링크, "테스트 관점에서 확인할 시나리오" 한 줄 요청. |
| **PM → 디자인** | 디자인 | 기능/화면 요약, 타깃 사용자, 제약(브랜드·접근성). |
| **디자인 → 프론트** | 프론트 웹/모바일 | 컴포넌트 규격 문서·Figma 링크, 토큰/변형 정의. (Figma MCP·Stitch 사용 시 링크만 있어도 됨) |
| **개발 → QA** | QA | MR/배포 브랜치 링크, "변경 범위 요약", 회귀 시 확인할 영역. |
| **QA → 인프라/배포** | 인프라 | 테스트 완료 요약, 배포 요청 이슈/체크리스트. |

- **슬랙 봇 활용**: Boss가 "스펙 확정됐어, 백엔드/프론트는 이 이슈 기준으로 진행해"라고 할 때, 봇이 **스펙 링크 + 위 표의 "PM → 개발" 항목**을 포맷에 맞춰 해당 채널에 올리면, 개발 에이전트에 "이 스펙 기준으로 plan 만들어줘" + 링크를 붙여 넣는 것만으로 일관된 입력이 됩니다.
- **에이전트 자체 고도화**: 각 포지션의 **AGENTS.md** 또는 **rules**에 "입력으로 받을 때: 스펙 링크/이슈 번호가 있으면 그걸 기준으로 plan·task 생성" 같은 한 줄 가이드를 두면, 슬랙에서 넘어온 문구+링크만으로도 에이전트가 올바르게 동작하도록 유도할 수 있습니다.

### 5.3 Boss가 슬랙 봇으로 업무를 유기적으로 진행시키는 경우

- **포지션별 채널에 "할 일 + 프롬프트 템플릿" 전달**: Boss가 지시한 내용을 봇이 **포지션별 PROMPT_TEMPLATES.md**에서 가져온 "자주 쓰는 프롬프트"와 결합해 해당 채널에 올립니다. (예: #proj-pm → "이번 스프린트 [기능명] 스펙 초안 작성. Cursor에서: '스펙 초안 만들어줘. [기능명]에 대해 목표·요구·AC 포함해줘.'")  
- **핸드오프 포맷으로 다음 포지션 알림**: PM 스펙 완료 시 봇이 "스펙 링크 + PM→개발 핸드오프 포맷"을 #proj-backend, #proj-frontend에 보내면, 각 포지션 담당자는 동일한 맥락으로 Cursor 에이전트를 사용할 수 있습니다.  
- **에이전트 고도화 요약**: (1) 각 포지션에 **PROMPT_TEMPLATES.md** + **`/start`** 로 "역질문" 지원 → 포지션을 모르는 사용자도 옵션 선택만으로 전문가급 요청 가능. (2) 슬랙 봇이 **지시 문구 템플릿**과 **핸드오프 포맷**을 역할 정의 레포에서 참고해 채널에 전달 → 포지션별 에이전트가 받을 입력이 일관되고, 팀 전체가 유기적으로 움직이게 됩니다.

### 5.4 이벤트 → 슬랙 메시지 템플릿 (Phase 2a)

이벤트 발생 시 슬랙에 올릴 **메시지 템플릿** 예시는 **[docs/AGENT_MESSAGE_PROTOCOL.md](AGENT_MESSAGE_PROTOCOL.md) §6**에 정리되어 있습니다.  
- `spec_completed`: Design·FE·BE·QA 채널에 "스펙 확정됨" + 목표·스펙 링크·AC·우선순위·Cursor 붙여넣기용 문구.  
- `pr_opened`: QA 채널에 "PR 리뷰 요청" + PR URL·변경 요약·회귀 힌트·Cursor 문구.  
봇/오케스트레이터는 해당 이벤트 수신 시 위 문서의 템플릿과 역할–채널 매핑(§4)을 사용해 메시지를 포스팅하면 됩니다.

## 6. 한 줄 요약

- **각 포지션별 레포에서 플러그인 사용**: 이 레포의 **각 포지션 폴더**에 `.cursor-plugin`이 있으므로, **해당 폴더를 플러그인 소스로 지정**하면 됩니다. 실무 레포에서는 그 플러그인을 "로컬 폴더" 또는 마켓플레이스로 추가해 사용합니다.  
- **실무 레포 세팅**: **플러그인**으로 해당 포지션 폴더를 참조하거나, **복사**로 `.cursor/` + `AGENTS.md` (+ 필요 시 `.cursor-plugin/`)를 실무 레포에 넣으면 됩니다.  
- **Slack 봇과 유기적 연동**: 봇은 **역할 정의(이 레포)** 를 규칙·템플릿 소스로 쓰고, **실무 레포**는 **웹훅**으로만 이벤트를 받아 Slack에 알림·체크리스트를 뿌리면, 실제 IT 회사 팀 프로젝트처럼 기획 → 개발 → 리뷰 → QA → 배포 → Slack 공유가 한 흐름으로 이어집니다.  
- **포지션별 역질문·템플릿**: 각 포지션 폴더의 **PROMPT_TEMPLATES.md**와 **`/start`** 명령으로, "뭐 할 수 있어?"라고만 해도 에이전트가 옵션을 제시하고 역으로 질문해 전문가급 요청을 이끌어 냅니다. **Boss가 슬랙으로 지시할 때**는 봇이 해당 포지션의 자주 쓰는 프롬프트 템플릿과 **핸드오프 포맷**(§4)을 채널에 전달하면, 포지션별 에이전트가 유기적으로 동작합니다.
