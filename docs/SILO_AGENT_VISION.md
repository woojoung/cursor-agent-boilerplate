# AI 사일로 조직 시스템 비전 및 실행 가능성

> 이 문서의 목적: **"토스식 사일로 조직을 AI 에이전트로 재현하는 시스템"**을, 현재 보일러플레이트(cursor-agent-boilerplate)를 기반으로 **실제로 구축해 나갈 수 있는지** 판단하고, 단계별 로드맵과 아키텍처 초안을 제시합니다.

---

## 1. 결론: 이대로 계속 가면 가능한가?

**가능합니다.**  
단, **현재 레포는 "Phase 1: 역할·워크플로우 정의 + 인간이 Cursor로 실행"**이고, **목표 비전은 "Phase 2~3: 슬랙 위에서 에이전트들이 스스로 대화·리뷰·에스컬레이션"**이므로, **추가로 구축할 레이어(오케스트레이션, 슬랙 에이전트 런타임, 메모리)**가 필요합니다.

이 레포를 **역할 정의·핸드오프·프롬프트의 단일 소스**로 유지하고, **별도 서비스(슬랙 봇 + 에이전트 런타임)**가 이 레포의 내용을 읽어서 동작하도록 설계하면, 비전에 도달하는 경로가 명확해집니다.

## 2. 현재 보일러플레이트 ↔ 목표 비전 매핑

| 목표 비전 (Master Prompt) | 현재 보일러플레이트 | 정합성 |
|---------------------------|----------------------|--------|
| **FRIDAY (PO)** | `pm/` — 스펙·태스크 분해·리스크, PROMPT_TEMPLATES, `/start`, `/spec-template`, `/task-breakdown`, `/risk-check` | ✅ 역할·산출물 일치 |
| **Black Widow (Designer)** | `design/` — 컴포넌트 규격·디자인 리뷰·토큰, `/component-spec`, `/design-review` | ✅ 역할·산출물 일치 |
| **Spider-Man (FE)** | `frontend-web/`, `frontend-mobile/` — 스펙→plan→task, 리뷰, MR, 배포 | ✅ 역할·산출물 일치 |
| **Hulk (BE)** | `backend/` — 스펙→plan→task, API·DB·인증, 리뷰, MR | ✅ 역할·산출물 일치 |
| **Hawkeye (QA)** | `qa/` — 시나리오·테스트 계획·버그 템플릿, 머지 전 검증 | ✅ 역할·산출물 일치 |
| **인프라 (선택)** | `infra/` — plan·리뷰·런북 | ✅ 보너스로 이미 존재 |
| **Slack에서 서로 대화** | INTEGRATION.md: 봇이 채널에 지시·핸드오프 전달 → **현재는 인간이 Cursor에서 실행** | ⚠️ 다음 단계에서 "에이전트가 슬랙에 응답" 필요 |
| **에이전트 독립 메모리** | 없음 | 🔲 별도 설계·구현 |
| **PR 단위 작업 + CI/CD** | rules·commands에 MR/PR·리뷰 규칙 있음. **자동 생성/머지는 없음** | ⚠️ 오케스트레이션 레이어에서 GitHub API 호출 필요 |
| **블로커 에스컬레이션·자동 롤백** | 문서로 개념만 있음 | 🔲 오케스트레이션 레이어에서 구현 |

**요약**: **역할·워크플로우·산출물·핸드오프 형식**은 이미 이 레포에 정의되어 있어, "에이전트 정의"와 "구조화된 소통 형식"의 **단일 소스**로 쓸 수 있습니다. 부족한 것은 **슬랙 위에서 에이전트가 직접 행동하는 런타임**과 **상태·메모리·에스컬레이션 로직**입니다.

## 3. 아키텍처 관점: 현재 vs 목표

### 3.1 현재 (Phase 1)

```
[인간] ←→ [Cursor IDE] ←→ [역할별 rules/commands/skills/AGENTS.md]
                ↑
[Slack 봇] ——→ 채널에 "할 일 + 프롬프트 템플릿" 전달 → 인간이 Cursor에 붙여넣기
```

- **에이전트 실행 주체**: 인간(개발자/기획/디자이너 등).
- **Slack**: 알림 + "이걸 해줘" 문구 전달. 에이전트는 Cursor 안에서만 동작.

### 3.2 목표 (Phase 2–3): 슬랙 기반 에이전트 협업

```
[인간: 전략·최종 승인만]

[Slack] ←→ [오케스트레이터] ←→ [역할별 에이전트 런타임]
   │              │                        │
   │              ├─ 이 레포(pm/, design/, backend/, …)의
   │              │  rules·AGENTS.md·PROMPT_TEMPLATES.md 로드
   │              │
   │              ├─ GitHub API (브랜치·PR·CI 상태)
   │              └─ 상태: TODO / IN_PROGRESS / BLOCKED / DONE
   │
   └─ 스레드별 대화, 리뷰 요청, 에스컬레이션, 배포 알림
```

- **에이전트 실행 주체**: 오케스트레이터가 이벤트(슬랙 메시지, GitHub 웹훅)에 따라 **해당 역할 에이전트**를 호출.
- **역할 에이전트**: 이 레포의 해당 포지션 폴더 내용(rules, AGENTS.md, PROMPT_TEMPLATES, 핸드오프 형식)을 **시스템 프롬프트·컨텍스트**로 사용.
- **Slack**: 에이전트가 **직접** 스레드에 답변·리뷰·블로커 신고를 올리는 "협업 공간".

## 4. 목표 달성을 위해 필요한 것 (이 레포 + 신규 구축)

### 4.1 이 레포에서 유지·강화할 것

| 항목 | 설명 |
|------|------|
| **역할 정의** | pm, design, backend, frontend-web, frontend-mobile, qa, infra — 그대로 FRIDAY·Black Widow·Spider-Man·Hulk·Hawkeye에 대응. |
| **역할별 프롬프트 엔진 소스** | 각 포지션의 `.cursor/rules`, `AGENTS.md`, `PROMPT_TEMPLATES.md`, commands — **에이전트 런타임이 여기서 프롬프트를 로드**하도록 설계. |
| **핸드오프 포맷** | INTEGRATION.md §4.2의 "이전 포지션 → 다음 포지션" 표 — **에이전트 간 메시지 프로토콜**의 스키마로 사용. |
| **워크플로우 단계** | spec → plan → task → 구현 → 리뷰 → MR → 배포 — 이미 명령·규칙으로 정의됨. 오케스트레이터가 **상태 전이**로 재사용. |

### 4.2 새로 설계·구축할 것

| 항목 | 설명 |
|------|------|
| **에이전트 런타임 (서비스)** | Cursor 밖에서 동작. Slack 이벤트/오케스트레이터 호출 수신 → 해당 역할의 프롬프트(이 레포에서 로드) + LLM 호출 → 산출물 생성·Slack 전송·GitHub API 호출. |
| **Slack 앱·봇** | 스레드 기반 대화, 멘션, 리마인드, 에스컬레이션(예: PO 호출). 봇이 "에이전트 대리"로 메시지 전송. |
| **오케스트레이터** | 작업 생성·할당·상태(TODO/IN_PROGRESS/BLOCKED/DONE) 관리. "스펙 완료" → "디자인 태스크 생성" 등 **전이 로직**. 타임아웃 시 리마인드·블로커 시 PO 호출. |
| **에이전트 메모리** | 역할별 컨텍스트 저장소(벡터 DB 또는 키-값). 대화·산출물·결정 요약 저장. 메모리 스키마·용도: **[docs/AGENT_MEMORY_SCHEMA.md](AGENT_MEMORY_SCHEMA.md)**. |
| **GitHub 연동** | 브랜치 생성, 커밋(에이전트가 생성한 코드/문서 반영), PR 생성, CI 상태 구독. 머지 차단은 QA 역할 규칙(이 레포) + 오케스트레이터가 "테스트 실패 시 머지 금지"로 구현. |
| **CI/CD·배포·롤백** | 배포 실패 시 자동 롤백 + Slack 알림 — 오케스트레이터 또는 별도 파이프라인 서비스. |

## 5. 에이전트 간 메시지 프로토콜 (초안)

이 레포의 **핸드오프 포맷**을 기반으로, 에이전트 간 구조화 메시지 예시입니다. 실제 구현 시 JSON 스키마로 고정하면 됩니다.

| 이벤트 | 발신 역할 | 수신 역할 | 페이로드 예시 |
|--------|-----------|-----------|----------------|
| `spec_completed` | PO | Designer, FE, BE, QA | `{ spec_id, spec_link, goal_one_liner, ac_list, priority }` |
| `design_completed` | Designer | FE | `{ design_spec_link, figma_link, tokens, variants }` |
| `task_ready_for_impl` | PO | FE, BE | `{ task_id, spec_ref, ac_ref, branch_name }` |
| `pr_opened` | FE/BE | QA | `{ pr_url, branch, change_summary, regression_hint }` |
| `qa_approved` / `qa_blocked` | QA | FE/BE, Orchestrator | `{ pr_url, approved \| blocked, reason }` |
| `deploy_done` / `deploy_failed` | CI/CD | Orchestrator, Slack | `{ env, success, rollback_done }` |
| `blocked_escalation` | any | PO | `{ task_id, blocker_reason, from_role }` |

오케스트레이터는 위 이벤트를 Slack 스레드·GitHub 웹훅과 매핑하고, 해당 역할 에이전트를 호출할 때 **페이로드 + 이 레포의 해당 역할 프롬프트**를 넣어 주면 됩니다.

## 6. 단계별 로드맵 (실행 가능한 순서)

| 단계 | 내용 | 이 레포에서 할 일 | 신규 구축 |
|------|------|-------------------|-----------|
| **Phase 1 (현재)** | 인간이 Cursor에서 역할별 에이전트 사용. Slack 봇은 "할 일 + 프롬프트"만 전달. | 유지. INTEGRATION.md, PROMPT_TEMPLATES, /start 유지·보완. | — |
| **Phase 2a** | Slack 대화창에서 **@Cursor**(Cloud Agent) 호출로 "할 일" 지시 → 결과를 슬랙으로 수신. 필요 시 [AGENT_MESSAGE_PROTOCOL](AGENT_MESSAGE_PROTOCOL.md) §6 템플릿을 복사해 붙여넣고 @Cursor 호출. 별도 봇 코드 없음. | 메시지 프로토콜·역할-채널 매핑 명시. **Cursor Slack 연동** 가이드: [PHASE2A_CURSOR_SLACK.md](PHASE2A_CURSOR_SLACK.md). | Cursor Dashboard에서 Slack 연동 설정. |
| **Phase 2b** | 오케스트레이터 서비스 도입. **상태만** 관리(TODO/IN_PROGRESS/DONE). 작업 할당·완료는 여전히 인간이 Cursor에서 수행. | 각 역할의 "완료 시 올려야 할 산출물·형식"을 AGENTS.md 또는 규칙에 한 줄씩 명시. | 오케스트레이터: 상태 저장, Slack 알림, (선택) GitHub 이슈 상태 연동. |
| **Phase 3a** | **한 역할만** 슬랙 에이전트화(예: PO). PO 봇이 슬랙 메시지 수신 → 이 레포의 `pm/` 프롬프트 로드 → LLM 호출 → 스레드에 답변·스펙 초안 링크. | `pm/` 아래에 "슬랙 봇용 시스템 프롬프트 요약" 또는 별도 `pm/slack-agent-prompt.md` 제공. | 에이전트 런타임 1개(PM 전용), Slack Event API, 간단한 메모리(스레드별 컨텍스트). |
| **Phase 3b** | 여러 역할을 슬랙 에이전트로 확장. PR 생성은 아직 인간 또는 반자동. | 역할별 `slack-agent-prompt.md` 또는 오케스트레이터가 읽을 **역할별 진입점** 문서. | 역할별 에이전트 런타임, 메시지 프로토콜 (§5) 구현. |
| **Phase 3c** | PR 자동 생성, CI 연동, QA 머지 차단, 배포 실패 시 롤백 + 알림. | 리뷰·머지 기준을 qa/ rules에 명시. 오케스트레이터가 "QA 승인 시에만 머지" 로직 참고. | GitHub API 연동, CI 웹훅, 롤백 정책. |

## 7. 핵심 철학과의 일치

> "AI는 개인의 업무를 대체하는 것이 아니라, **조직 구조를 재현**해야 한다."

이 레포는 이미 **조직 구조(역할·산출물·핸드오프)**를 문서와 프롬프트로 재현하고 있습니다.  
**Cursor + 인간**은 "1인 다역할"로 그 구조를 실행하는 형태이고,  
**슬랙 에이전트 런타임**은 같은 구조를 "역할별 에이전트가 슬랙에서 협업"하도록 실행하는 형태입니다.  
따라서 **이 레포를 단일 소스로 두고**, 오케스트레이션·런타임·메모리만 별도 구축하면, 비전과의 일치도를 유지한 채 확장할 수 있습니다.

## 9. 요약
