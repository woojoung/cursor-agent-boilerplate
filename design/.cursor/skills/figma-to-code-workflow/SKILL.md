# Figma → 코드 구현 워크플로

참고 문헌: [피그마&클로드 실무 치트키들 (Figmapedia)](https://figmapedia.com/entry/327fdea8-0034-8084-992e-ced9b4e29ae1) — 디자인 구현·토큰·접근성·시각화·UX 라이팅·상황별 명령 보강용.

전체 프로세스·채널 한계: [docs/process/figma-mcp-handoff.md](../../../docs/process/figma-mcp-handoff.md)

## 핵심 원칙 (Figmapedia 요지)

- **실시간 브라우저 UI**로 보는 방식과 **소스/레포·Figma 노드**를 분석하는 방식은 목적이 다르다.
  - **브라우저/스크린**: “지금 이렇게 보인다”에 강함 → 시각적 검증·차이 잡기.
  - **노드·코드**: 구조·간격·컴포넌트 매핑·유지보수에 강함 → 구현의 진실 공급원은 **레포(토큰·컴포넌트) + Figma 구조**에 둔다.
- 실무에서 빠뜨리지 않을 축: **디자인 구현**, **토큰 동기화**, **접근성 검사**, **시각화(비교)**, **UX 라이팅**.
- **“최고 퀄리티까지 무한 자동 루프”는 하지 않는다.** 루프는 **종료 조건 + 최대 반복 횟수**가 있을 때만 수행한다.

---

## 사용자가 막혔을 때 (에이전트 행동)

1. **지금 어느 단계인지** 한 줄로 말한다: 전제 / 읽기 / 구현 / 검증(또는 자동 루프).
2. **다음 한 가지 행동만** 요청한다.
3. 아래 표로 **빠진 전제**를 진단·질문한다.

| 증상 | 먼저 확인 |
|------|-----------|
| MCP가 안 됨 | Figma 플러그인 실행, `join_channel` 채널 문자열 일치 |
| 구현이 들쭉날쭉 | 범위가 **프레임 단위**인지, 전체 파일인지 |
| 색·간격만 틀림 | 노드 `get_node_info` 등으로 수치 재확인 + 토큰 매핑 |
| “디자인과 다르다”만 반복 | 스크린샷 1장과 **차이 목록**을 글머리로 먼저 쓰기 |
| 문구가 어색함 | UX 라이팅: 톤·길이·버튼 라벨 규칙을 사용자에게 확인 |

사용자가 “뭐부터?”라고 하면 **Phase 0 체크리스트 순서**로 질문한다.

---

## Phase 0 — 전제 (구현 전 필수)

- [ ] **범위**: 구현할 **프레임/플로우 이름** 확정 (**전체 파일 덤프 금지**).
- [ ] **뷰포트**: 데스크톱 폭, 브레이크포인트 등 **한 줄 이상** 명시.
- [ ] **연결**: Talk to Figma 사용 시 **`join_channel`** 완료 (채널은 사용자가 플러그인에서 복사 — AI가 자동 조회 불가).
- [ ] **진실 공급원**: 레포의 **디자인 토큰·공용 컴포넌트** 경로 @ 참조 또는 사용자 지정.
- [ ] **토큰 동기화**: Figma 변수/스타일 이름 ↔ 코드 토큰 어휘 정렬; 불일치면 매핑 표 요청 또는 함께 작성.

MCP로 **레이어/노드 이름 일괄 변경**은 Talk to Figma 도구 세트에 일반적으로 없음 → Figma에서 수동·디자인 작업.

---

## Phase 0-B — MCP 전제 점검 (`/figma-mcp-precheck`)

- 채널 문자열은 **매 연결마다 갱신될 수 있다.** 사용자가 채팅에 붙여넣지 않으면 `join_channel` 호출 전에 요청한다.
- 절차: `join_channel` → `get_document_info` → `get_styles` → `get_local_components` → 레포 경로와 이름 갭 표.
- 상세: 명령 `figma-mcp-precheck.md` 참고.

---

## Phase 1 — 읽기 (구조 + 시각)

**병렬로** 모은다.

1. **구조**: `read_my_design` 또는 선택 프레임 노드 트리 (위계·오토레이아웃·텍스트·인스턴스명).
2. **시각(비교 기준)**: `export_node_as_image` 또는 동등 스크린샷 **1장** — Phase 3·자동 루프의 기준으로 사용.
3. **메타**: 필요 시 `get_styles`, `get_local_components`.

**금지**: 의미 없이 전체 문서를 한 번에 긁어 구현 요청하기.

---

## Phase 2 — 구현

1. **DS 우선**: 기존 컴포넌트·토큰에 매핑; 없을 때만 최소 단위 신규 생성.
2. **레이아웃**: 오토레이아웃 → flex/grid; 숫자는 가능하면 **토큰/스케일**로.
3. **UX 라이팅**: 카피는 placeholder 또는 사용자에게 톤·길이 제한 확인.
4. **접근성**: 제목 단계, 버튼 vs 링크, 라벨, 포커스 — 노드만으로 부족하면 요구사항 한 줄이라도 확인.

**컴포넌트화**: 반복 UI는 Figma·코드 모두 컴포넌트 단위로 맞추면 구현 정합성과 수정 루프에 유리하다.

---

## Phase 3 — 검증 루프 (수동·기본)

1. **시각**: Phase 1 스크린샷과 구현 화면 비교 → **차이 목록**만 수정.
2. **수치**: 어긋난 노드만 `get_node_info` / `get_nodes_info`로 좁혀 수정.
3. **토큰**: 하드코딩 색·간격 잔존 여부 점검.
4. **접근성**: 스모크(포커스, 가능하면 대비).

### 흔한 실수 → 대응

- 노드만 보고 끝냄 → Phase 3 시각 비교 필수 안내.
- 픽셀만 맞춤 → 반응형·토큰 규칙 질문.
- 레이어 이름이 `Frame 123` 수준 → 리네이밍 또는 annotations 요청; **전 노드 완벽 네이밍은 비현실적** — 섹션·컴포넌트 경계 우선.
- 브라우저 캡처만 믿음 → 구조·토큰은 레포 + Figma 노드로 보강 안내.

---

## 에이전트 자동 검증 루프 (선택)

사용자가 **“루프 돌려 줘”** 등으로 명시했을 때만 적용. **주관적 ‘완벽’이 아니라 아래 종료 조건**으로만 반복한다.

### 고정 규칙

1. **최대 반복 횟수**: 기본 **3회**. 사용자가 숫자를 주면 그만큼. 초과 시 멈추고 **남은 차이 목록**을 사용자에게 넘긴다.
2. **한 라운드당 순서 (고정)**  
   - 구현 화면을 확인 가능한 상태로 둔다 (dev 서버·캡처 등 프로젝트에 맞게).  
   - Phase 1의 **Figma 스크린샷/export**와 비교해 **차이 목록**을 글머리로 작성.  
   - 차이마다 **한 번에 하나의 논리적 수정** (파일/컴포넌트 단위로 묶기).  
   - **객관적 검증**: 빌드·린트·프로젝트에 정의된 테스트가 있으면 실행해 통과 확인.
3. **종료 조건 (하나라도 해당 시 루프 종료)**  
   - 차이 목록이 **비었음**.  
   - 객관적 검증이 **모두 통과**하고, 스크린 비교 잔여 차이가 **의도적/토큰 한계**로 사용자가 동의한 항목만 남음.  
   - **최대 반복 횟수 도달**.
4. **금지**  
   - 종료 조건·상한 없이 “만족할 때까지” 무한 반복.  
   - 같은 파일에 **의미 없는 미세 조정만** 반복: **2회 연속** diff가 거의 없으면 사용자에게 승인 요청 후 종료.

### 루프 시작 전 사용자에게 확인 (없으면 질문)

- 최대 반복 횟수 (기본 3).  
- “통과” 최소 기준: 예) `./gradlew build` / `pnpm build` / 나열한 체크리스트.  
- 비교 뷰가 여러 개(모바일·다크 등)면 **우선순위 1개**만 첫 루프에 포함.

## Talk to Figma MCP 도구 참고

- 연결: `join_channel`

- 시각: `export_node_as_image`  
- (전체 목록은 프로젝트 MCP 설정 기준)
